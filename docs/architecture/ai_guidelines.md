# ZigCel AI Developer Instructions

このファイルは、ZigCelプロジェクトに参加するAIエージェントのためのシステムプロンプト・ガイドラインです。
セッションが新しくなったり、AIエージェントが変わったりした場合でも、このファイルを読むことでプロジェクトの前提条件をシームレスに引き継ぐことができます。

## プロジェクト概要
**ZigCel** は、あらゆるWebフレームワーク環境（React, Vue, Rails ERB 等）でシームレスに動作する、超高速なスプレッドシート・コンポーネントです。

## 技術スタックとアーキテクチャ
本プロジェクトは以下の3つの層から構成されるアーキテクチャを採用しています。

1. **コアエンジン (Zig / WASM)**:
   - 言語: Zig (v0.13.0)
   - 役割: データ構造の管理、数式パース、依存関係グラフに基づく再計算処理。
   - 制約: 描画やDOM操作には一切関与せず、純粋なC/WASMライブラリとして振る舞うこと。
2. **UIレンダリング層 (TypeScript + HTML5 Canvas)**:
   - 役割: WASMから計算結果を受け取り、Canvas APIを用いて高速に画面へ描画する。
   - 制約: セル1つ1つをDOM要素（div等）として描画してはいけない。パフォーマンスのために必ずCanvasに直接描画すること。
3. **バインディング層 (Web Components)**:
   - 役割: 定義済みの `<zig-cel></zig-cel>` カスタム要素を用いて、各フレームワークから呼び出せるようにするラッパー。

## 開発環境 (Docker Compose)
開発はmacOS環境を想定していますが、Node.jsやZigのバージョン差異を吸収するため、Docker Compose上で完結させます。
- コンテナの起動: `docker compose up -d`
- Zigのコンパイル: `docker compose exec app zig build`
- 開発サーバー: `http://localhost:5173` でHMR対応のViteサーバーが稼働します。

### テストフレームワークの実行
常にテストを書いて実行する習慣（TDD）を徹底してください。
- Zigのテストコンテナ内実行: `docker compose exec app zig build test`
- フロントエンド(TS)のテストが必要になった場合は適宜設定し `npm run test` 等で実行する。

## AIエージェントへの指示（ルール）
- **コンテキストの保持**: 作業を開始する前に、必ず `docs/tasks/current_tasks.md` と `docs/history/` 以下の直近の機能要件を読み、現在の進捗と直前の実装状況を把握してください。
- **タスクの更新**: 新しいタスクを開始・完了した際は、必ず `docs/tasks/current_tasks.md` にチェック(`[x]`)を入れ、タスクごとにさらに階層化されたドキュメントがあればそれを更新してください。
- **TDDと細かいコミット**: 意味のある最小単位でタスクを細分化して開発を進めてください。コードを変更したら「必ず」対応するテストコードを書き、テストを通し、グリーンになった段階ですぐにGitへコミットしてください。数時間分の作業をまとめてコミットするのは禁止です。
- **履歴の保存**: マイルストーン（Step）が完了するごとに、何を作ったか、何をテストしたかを `docs/history/` ディレクトリにMarkdown形式で保存してください。
- **言語**: コード内のコメントは英語でも構いませんが、コミットメッセージや各種ドキュメント、ユーザーとのコミュニケーションは**必ず日本語**で行ってください。
